# subagent-driven-development 技能详解 - 子代理驱动开发流程

> 本文档详细描述 subagent-driven-development 技能如何通过子代理执行实现计划

---

## 一、核心概念

### 1.1 什么是子代理驱动开发？

**定义：** 为每个任务派发独立的子代理执行，每个任务完成后进行双阶段审查（规范符合性审查 + 代码质量审查）。

**核心原则：**
```
新子代理 + 双阶段审查 = 高质量、快速迭代
```

### 1.2 为什么使用子代理？

| 优势 | 说明 |
|------|------|
| **隔离上下文** | 子代理不继承会话历史，避免上下文污染 |
| **精确指令** | 控制者精确构造子代理需要的上下文 |
| **专注执行** | 子代理专注于单一任务，不会被干扰 |
| **保留控制者上下文** | 控制者上下文用于协调工作，不被任务细节占用 |

### 1.3 与其他执行方式的对比

| 方式 | 特点 | 适用场景 |
|------|------|----------|
| **手动执行** | 人工逐步执行 | 小任务、需要频繁决策 |
| **executing-plans** | 当前会话批量执行 | 无子代理支持、需要并行会话 |
| **subagent-driven-development** | 子代理 + 双阶段审查 | 有子代理支持、任务相对独立 |

## 三、各阶段详细说明

### Step 1: 读取计划并提取任务

**目的：** 提取所有任务及其上下文，为子代理准备完整信息

**操作步骤：**
```
1. 读取计划文件: docs/plans/YYYY-MM-DD-<feature-name>.md

2. 提取所有任务（包含完整内容）:
   - Task 1: [任务名称]
     - Files: [文件列表]
     - Steps: [所有步骤]
     - 代码示例

3. 记录上下文信息:
   - 计划目标 (Goal)
   - 架构方法 (Architecture)
   - 技术栈 (Tech Stack)

4. 创建 TodoWrite 任务列表:
   - Task 1: [pending]
   - Task 2: [pending]
   - Task N: [pending]
```

**重要：** 必须提取任务的**完整文本**，而不是让子代理去读取文件

**为什么？**
- 子代理不应该读取计划文件（避免上下文污染）
- 控制者精确提供子代理需要的内容
- 提高效率（子代理直接获得信息）

---

### Step 2: 设置工作环境

**目的：** 确保在隔离的工作环境中执行

**检查项：**
```
1. 确认在 worktree 中工作
   - 如果不在，调用 using-git-worktrees 创建

2. 记录基础 commit SHA
   - 用于后续代码审查的 diff 比较

   git rev-parse HEAD → BASE_SHA
```

---

### Step 3: 派发实现子代理

**目的：** 让子代理执行具体任务

#### 3.1 子代理提示模板

```
Task tool (general-purpose):
  description: "Implement Task N: [任务名称]"
  prompt: |
    You are implementing Task N: [任务名称]

    ## Task Description

    [任务全文 - 从计划中粘贴，不让子代理读取文件]

    ## Context

    [场景设置：任务位置、依赖关系、架构上下文]

    ## Before You Begin

    如果你对以下内容有疑问：
    - 需求或验收标准
    - 实现方法或策略
    - 依赖关系或假设
    - 任务描述中不清楚的内容

    **现在提问。** 在开始工作前提出任何顾虑。

    ## Your Job

    一旦需求明确：
    1. 实现任务指定的内容
    2. 编写测试（如果任务要求 TDD）
    3. 验证实现工作正常
    4. 提交工作
    5. 自我审查
    6. 报告结果

    工作目录: [目录路径]

    ## Report Format

    完成后报告：
    - **Status:** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么
    - 测试内容和结果
    - 修改的文件
    - 自我审查发现（如有）
    - 任何问题或顾虑
```

#### 3.2 处理子代理状态

| 状态 | 含义 | 处理方式 |
|------|------|----------|
| **DONE** | 完成工作 | 进入规范审查阶段 |
| **DONE_WITH_CONCERNS** | 完成但有顾虑 | 阅读顾虑内容，判断是否需要先处理 |
| **NEEDS_CONTEXT** | 需要更多信息 | 提供缺失信息，重新派发 |
| **BLOCKED** | 无法完成 | 评估阻塞原因，升级或调整 |

#### 3.3 处理子代理提问

**原则：** 子代理提问时，必须回答后再让其继续

```
实现子代理: "开始前有个问题 - 密码应该用什么加密方式？"

控制者: "使用 BCrypt 加密，强度 10"

实现子代理: "明白了，开始实现..."
```

**为什么重要？**
- 避免子代理猜测导致错误
- 确保实现符合预期
- 在工作开始前对齐理解

---

### Step 4: 规范符合性审查

**目的：** 验证实现者构建了正确的内容（不多不少）

**触发时机：** 实现子代理报告 DONE 或 DONE_WITH_CONCERNS

#### 4.1 审查子代理提示模板

```
Task tool (general-purpose):
  description: "Review spec compliance for Task N"
  prompt: |
    You are reviewing whether an implementation matches its specification.

    ## What Was Requested

    [任务需求全文]

    ## What Implementer Claims They Built

    [来自实现者报告]

    ## CRITICAL: Do Not Trust the Report

    实现者完成得可疑地快。他们的报告可能不完整、不准确或过于乐观。
    你必须独立验证一切。

    **不要：**
    - 相信他们实现了什么
    - 相信他们的完整性声明
    - 接受他们对需求的解释

    **必须：**
    - 阅读他们写的实际代码
    - 逐行对比实际实现与需求
    - 检查他们声称实现但缺失的部分
    - 查找他们没提到的额外功能

    ## Your Job

    阅读实现代码并验证：

    **缺失的需求：**
    - 是否实现了所有请求的内容？
    - 是否有跳过或遗漏的需求？
    - 是否声称实现了某功能但实际没有？

    **多余的工作：**
    - 是否构建了未请求的内容？
    - 是否过度工程化或添加了不必要功能？
    - 是否添加了规范中没有的"锦上添花"功能？

    **误解：**
    - 是否对需求的理解与预期不同？
    - 是否解决了错误的问题？
    - 是否实现了正确的功能但方式错误？

    报告格式：
    - ✅ 规范符合（代码检查后一切匹配）
    - ❌ 发现问题：[具体列出缺失或多余的内容，包含 file:line 引用]
```

#### 4.2 审查结果处理

```
┌─────────────────────┐
│ 规范审查结果         │
└──────────┬──────────┘
           │
    ┌──────┴──────┐
    │             │
    ▼             ▼
  ✅ 通过      ❌ 有问题
    │             │
    │             ▼
    │    ┌─────────────────────┐
    │    │ 实现者修复问题       │
    │    └──────────┬──────────┘
    │               │
    │               ▼
    │    ┌─────────────────────┐
    │    │ 重新派发规范审查     │
    │    └──────────┬──────────┘
    │               │
    │               └──────────┐
    │                          │
    │              ┌───────────┴───────────┐
    │              │ 直到 ✅ 通过          │
    │              └───────────┬───────────┘
    │                          │
    └──────────────────────────┘
               │
               ▼
          进入代码质量审查
```

**重要规则：**
- 规范审查发现问题 = 实现者修复 = 重新审查
- 不能跳过重新审查
- 不能"差不多就行"

---

### Step 5: 代码质量审查

**目的：** 验证实现构建得很好（干净、可测试、可维护）

**触发时机：** 规范符合性审查通过后

**重要：** 必须等规范审查通过后才能进行代码质量审查

#### 5.1 审查内容

| 维度 | 检查内容 |
|------|----------|
| **代码组织** | 每个文件职责清晰，接口定义良好 |
| **命名** | 名称清晰准确，反映功能而非实现 |
| **测试** | 测试验证行为，遵循 TDD，覆盖全面 |
| **YAGNI** | 只构建请求的内容，不过度工程 |
| **文件大小** | 新创建的文件是否过大 |

#### 5.2 审查子代理调用

```
Task tool (code-reviewer):
  WHAT_WAS_IMPLEMENTED: [来自实现者报告]
  PLAN_OR_REQUIREMENTS: Task N from [plan-file]
  BASE_SHA: [任务前的 commit]
  HEAD_SHA: [当前 commit]
  DESCRIPTION: [任务摘要]
```

#### 5.3 审查结果处理

审查者返回：
- **Strengths:** 优点
- **Issues:**
  - Critical（关键）：必须修复
  - Important（重要）：应该修复
  - Minor（次要）：可以修复
- **Assessment:** 评估结论

```
┌─────────────────────┐
│ 质量审查结果         │
└──────────┬──────────┘
           │
    ┌──────┴──────┐
    │             │
    ▼             ▼
  ✅ 通过      ❌ 有问题
    │             │
    │             ▼
    │    ┌─────────────────────┐
    │    │ 实现者修复问题       │
    │    └──────────┬──────────┘
    │               │
    │               ▼
    │    ┌─────────────────────┐
    │    │ 重新派发质量审查     │
    │    └──────────┬──────────┘
    │               │
    │               └──────────┐
    │                          │
    │              ┌───────────┴───────────┐
    │              │ 直到 ✅ 通过          │
    │              └───────────┬───────────┘
    │                          │
    └──────────────────────────┘
               │
               ▼
           任务完成
```

### Step 6: 标记任务完成

**操作：**
```
1. 更新 TodoWrite 状态为 completed

2. 记录当前 commit SHA 作为下一个任务的基础
   HEAD_SHA = git rev-parse HEAD
```

---

### Step 7: 最终代码审查

**目的：** 审查整体实现是否一致、完整

**触发时机：** 所有任务完成后

**审查内容：**
- 所有任务是否正确集成
- 整体代码风格是否一致
- 是否有遗漏的集成问题

---

### Step 8: 完成开发分支

**操作：** 调用 `finishing-a-development-branch` 技能

```
Skill tool:
  skill: "xo1997-dev:finishing-a-development-branch"
```

## 四、模型选择策略

**原则：** 使用能完成任务的最便宜模型，节省成本并提高速度

### 4.1 任务复杂度判断

| 复杂度 | 特征 | 推荐模型 |
|--------|------|----------|
| **低** | 1-2 个文件，完整规范，清晰步骤 | 快速/便宜模型 |
| **中** | 多文件协调，集成考虑，模式匹配 | 标准模型 |
| **高** | 架构决策，设计判断，广泛代码理解 | 最强模型 |

### 4.2 角色与模型匹配

| 角色 | 推荐模型 | 原因 |
|------|----------|------|
| **实现者（机械任务）** | 快速模型 | 计划完整时，实现是机械工作 |
| **实现者（集成任务）** | 标准模型 | 需要理解多文件关系 |
| **规范审查者** | 最强模型 | 需要准确判断是否符合规范 |
| **质量审查者** | 最强模型 | 需要识别代码质量问题 |

## 六、双阶段审查详解

### 6.1 为什么需要双阶段审查？

```
┌─────────────────────────────────────────────────────────────┐
│                    双阶段审查的目的                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  阶段 1: 规范符合性审查 (Spec Compliance Review)            │
│  ─────────────────────────────────────────────             │
│  问题: 构建了正确的内容吗？                                   │
│  检查:                                                      │
│    - 是否实现了所有需求？                                     │
│    - 是否有遗漏的功能？                                       │
│    - 是否添加了未请求的功能？                                 │
│                                                             │
│  阶段 2: 代码质量审查 (Code Quality Review)                  │
│  ─────────────────────────────────────                      │
│  问题: 构建得好吗？                                          │
│  检查:                                                      │
│    - 代码是否干净、可维护？                                   │
│    - 测试是否充分？                                          │
│    - 是否遵循最佳实践？                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 审查顺序的重要性

**必须先进行规范审查，再进行质量审查**

```
❌ 错误顺序：
质量审查 → 发现代码很干净 → 但实现了错误的功能

✅ 正确顺序：
规范审查 → 确认功能正确 → 质量审查 → 确认代码干净
```

### 6.3 审查循环规则

```
发现问题的处理流程:

审查者发现问题时:
  1. 不能忽略问题
  2. 不能"差不多就行"
  3. 必须让实现者修复
  4. 必须重新审查
  5. 直到审查通过才能继续

审查循环限制:
  - 无最大迭代次数限制
  - 但如果修复多次仍有问题，考虑:
    a. 任务拆分
    b. 提供更多上下文
    c. 升级到更强模型
```

## 七、常见问题处理

### Q1: 子代理报告 BLOCKED 怎么办？

**处理步骤：**
```
1. 阅读阻塞原因

2. 评估阻塞类型:
   - 上下文问题 → 提供更多信息，重新派发
   - 能力问题 → 使用更强模型重新派发
   - 任务过大 → 拆分为更小任务
   - 计划错误 → 升级给用户

3. 不要忽略或强制重试
```

### Q2: 规范审查多次不通过怎么办？

**处理步骤：**
```
1. 检查计划是否清晰
   - 需求是否明确？
   - 是否有歧义？

2. 检查子代理理解
   - 是否正确理解需求？
   - 是否需要更多上下文？

3. 考虑升级模型
   - 使用更强的模型审查
   - 使用更强的模型实现

4. 最后手段: 升级给用户
```

### Q3: 子代理提问太多怎么办？

**处理：**
```
1. 检查任务描述是否完整
   - 是否缺少关键信息？
   - 是否有歧义？

2. 提供更完整的上下文
   - 更新任务描述
   - 添加示例

3. 耐心回答问题
   - 问题总比错误实现好
```

### Q4: 可以并行派发多个实现子代理吗？

**回答：不可以**

```
❌ 错误: 同时派发多个实现子代理
- 可能修改相同文件
- 可能产生冲突
- 难以追踪问题

✅ 正确: 顺序执行任务
- 每个任务完成后审查
- 记录 commit SHA
- 再开始下一个任务
```

## 八、关键原则总结

| 原则 | 说明 |
|------|------|
| **新子代理** | 每个任务使用新的子代理，不继承上下文 |
| **完整上下文** | 控制者提供完整任务内容，不让子代理读取文件 |
| **回答问题** | 子代理提问时必须回答后再让其继续 |
| **双阶段审查** | 先规范审查，再质量审查 |
| **审查循环** | 发现问题 → 修复 → 重新审查 |
| **顺序执行** | 一个任务完成后再开始下一个 |
| **模型选择** | 根据任务复杂度选择合适模型 |

## 九、禁止事项

| 禁止 | 原因 |
|------|------|
| 跳过规范审查 | 可能实现错误功能 |
| 跳过质量审查 | 可能产生低质量代码 |
| 接受"差不多" | 审查发现问题 = 未完成 |
| 并行派发多个实现者 | 可能产生文件冲突 |
| 让子代理读取计划文件 | 上下文污染 |
| 忽略子代理问题 | 导致错误实现 |
| 跳过重新审查 | 问题可能未真正修复 |
| 在规范审查通过前进行质量审查 | 顺序错误 |
