# For Test - 测试人员工作流技能

## 概述

测试人员在 xo1997-dev 工作流中的参与入口，确保测试活动与开发流程紧密衔接。

## 触发时机

- 设计文档批准后，需要细化测试用例时
- 实现计划完成后，需要补充测试用例骨架时
- 开发提测后，需要执行验收测试时
- 功能上线前，需要出具测试报告时

## 测试人员职责

| 阶段 | 职责 | 输出 |
|------|------|------|
| 设计阶段 | 参与需求评审，确认验收标准 | 验收标准确认 |
| 设计阶段 | 审查设计文档，提出测试关注点 | 测试意见 |
| 实现阶段 | 补充完整测试用例 | 测试用例文档 |
| 实现阶段 | 执行测试，记录缺陷 | 缺陷报告 |
| 完成阶段 | 出具测试报告 | 测试报告（含签名） |

## 流程

### Phase 1: 测试用例设计

基于设计文档，补充完整测试用例：

```
输入：
- docs/specs/feature_{模块}_{功能}_{日期}/design.md
- docs/specs/feature_{模块}_{功能}_{日期}/plan.md

输出：
- docs/specs/feature_{模块}_{功能}_{日期}/test-cases.md
```

**设计内容：**
1. 功能测试用例
2. 边界测试用例
3. 异常测试用例
4. UI 自动化用例（如适用）

**测试用例原则：**
- 每个用例可独立执行
- 步骤清晰，数据明确
- 预期结果可验证

### Phase 2: 测试用例评审

**评审参与方：**
- 测试负责人（主持）
- 开发负责人
- 产品/需求方（如涉及验收标准）

**评审内容：**
- 边界条件是否覆盖完整
- 异常场景是否考虑周全
- 验收标准是否与需求一致
- 用例是否可执行

**输出：**
- 更新后的测试用例文档
- 评审签署

### Phase 3: 测试执行

**执行顺序：**
```
1. 冒烟测试 → 核心功能是否可用
       ↓
2. 功能测试 → 功能是否符合设计
       ↓
3. 边界测试 → 边界条件处理
       ↓
4. 异常测试 → 异常场景处理
       ↓
5. 回归测试 → 修复是否引入新问题
```

**执行记录：**
- 记录每个用例的执行结果
- 缺陷提交到缺陷管理系统
- 阻塞用例标记原因

### Phase 4: 测试报告

**报告内容：**
```
## 测试概要
- 测试范围
- 测试环境
- 测试执行时间

## 测试执行汇总
- 用例总数、通过数、失败数、阻塞数
- 通过率

## 缺陷分析
- 缺陷总数、等级分布
- 缺陷类型分布
- 遗留问题及处置

## 测试结论
- 是否可以上线
- 上线条件确认

## 测试签名
| 角色 | 姓名 | 日期 |
|------|------|------|
| 测试负责人 | | |
| 产品负责人 | | |
```

**输出：**
- `docs/specs/feature_{模块}_{功能}_{日期}/test-report.md`

## 与其他技能的协作

```
for-product-owner (需求确认)
        │
        │ 需求文档 + 验收标准
        ▼
brainstorming (设计评审)
        │
        │ 设计文档 + 测试关注点
        ▼
writing-plans (实现计划)
        │
        │ 实现计划 + 测试用例骨架
        ▼
for-test (测试活动)
        │
        │ 测试用例 + 测试执行
        ▼
verification-before-completion (验证)
        │
        │ 验证报告
        ▼
上线 / 完成
```

## 测试用例模板

使用标准模板：`docs/templates/test-case-template.md`

## 注意事项

1. **测试左移**：尽早参与设计，提前发现需求问题
2. **缺陷跟踪**：所有缺陷必须录入缺陷管理系统
3. **可复现**：每个缺陷必须有明确的复现步骤
4. **闭环**：修复后必须验证关闭

- [ ] 所有缺陷已提交并跟踪
- [ ] 遗留问题已确认处置方式
- [ ] 测试报告已出具并签署
